# Пайплайн обработки ответов UMUX-Lite

Бэкенд-пайплайн, который принимает сырые выгрузки опросника UMUX-Lite (2 пункта,
шкала 1–5) заданной схемы, очищает и валидирует их, считает UMUX-скор по каждой
анкете и формирует наглядную сводку по продуктам, версиям и времени — без
необходимости лезть в код.

```
umux_score = ((score1 - 1) + (score2 - 1)) / 8 * 100
```

## Почему данные нужно чистить

В образце выгрузки (`sample_data/raw_responses_sample.csv`, 1560 строк) присутствует
типичный набор дефектов «сырой» таблицы, в которую писали разные, неконтролируемые
источники:

- **Опечатки/разный регистр в категориальных полях**: `product` встречается как
  `Serch`, `Onboardng`, `Check out`, `CHECKOUT`, `checkout`, ` Checkout ` — и это явно
  один и тот же продукт из пяти возможных.
- **Непоследовательный регистр/пробелы** в `platform` (`web`/`Web`/`ANDROID`),
  `country` (`ru`/`RU`), `user_segment` (`New`/`new`/`returning `).
- **Пропуски**: `platform` (19%), `country` (13%), `user_segment` (30%),
  `submitted_at` и оценки (несколько процентов) — пустые.
- **Оценки вне диапазона / мусор**: `0`, `-1`, `6`, `7`, `99` при шкале 1–5.
- **Битые даты**: нераспознаваемые строки, невозможные даты (`2024-13-40 99:99`).
- **Дубликаты**: точные повторы строк, а также строки с одинаковым `response_id`, но
  разным содержимым (например, повторная отправка с исправлением).

## Правила очистки (и почему именно так)

Общий принцип: **отбраковывать строку только тогда, когда пропущенное/некорректное
поле реально нужно для расчёта скора или для агрегации, которую требует задание
(продукт / версия / время).** Всё остальное сводится к бакету `"unknown"`, а не
выбрасывается — потому что терять ~20% строк из-за пустой ячейки `country` хуже для
продакта, чем увидеть там пометку «неизвестно».

| Поле | Обработка |
|---|---|
| `response_id` | обрезка пробелов; пусто → отбраковка (`missing_response_id`) |
| `submitted_at` | парсится как дата-время; не распознано или год вне `[2015, 2100]` → отбраковка |
| `product` | нормализация (пробелы/регистр) + **fuzzy-сопоставление** с настраиваемым списком канонических значений (`Checkout`, `Payments`, `Search`, `Onboarding`, `Profile`); нет уверенного совпадения → отбраковка (`unrecognized_product`), т.к. «агрегация по продукту» без продукта невозможна |
| `product_version` | только обрезка пробелов; пусто → отбраковка. Канонического списка нет — версии не фиксированный перечень |
| `platform` | fuzzy-сопоставление с `Web`/`Android`/`iOS`; пропуск/несовпадение → `"unknown"`, строка остаётся |
| `country` | приведение к верхнему регистру; пропуск → `"unknown"`, строка остаётся |
| `user_segment` | fuzzy-сопоставление с `new`/`returning`/`power`; пропуск/несовпадение → `"unknown"`, строка остаётся |
| `score1`, `score2` | должно парситься как целое число в `[1, 5]`; пропуск/не целое/вне диапазона → отбраковка |
| дубликаты | точный повтор всей строки → оставляем первую, остальные отбраковываем (`exact_duplicate_row`); одинаковый `response_id` с разным содержимым → оставляем первую валидную, остальные отбраковываем (`conflicting_duplicate_response_id`) |

Списки «известных» продуктов/платформ/сегментов — это **конфигурация, а не зашитая в
код логика** (`umux_pipeline/config.py`, переопределяется через `--config
some.json`): fuzzy-сопоставление означает, что новый источник с чуть другими
опечатками всё равно нормализуется без правки кода, а по-настоящему новое название
продукта будет честно помечено, а не тихо занесено не туда.

Каждая отбракованная строка попадает в `output/rejected_rows.csv` с указанием
`reject_reason`, причины логируются (с количеством) в `output/pipeline.log` и в
консоль. Повторный запуск пайплайна перезаписывает папку с результатами, а не
дописывает в неё — поэтому повторные запуски на одних и тех же входных данных
идемпотентны.

## Что делает пайплайн

1. Загружает и объединяет один или несколько входных CSV-файлов заданной схемы.
2. Нормализует и валидирует (правила выше) → делит на `valid` / `rejected`.
3. Считает `umux_score` по каждой валидной анкете.
4. Строит агрегаты:
   - средний/медианный UMUX по продукту
   - средний/медианный UMUX по продукту и версии
   - средний UMUX по месяцам и продуктам (динамика)
   - средний UMUX по сегменту пользователей
   - количество/доля отбракованных строк по причинам
5. Формирует самодостаточный `report.html` (графики встроены как base64 PNG, без
   внешних ресурсов) и CSV/JSON-файлы для дальнейшего анализа.

### Схема входных данных

| Поле | Описание |
|---|---|
| `response_id` | идентификатор анкеты |
| `submitted_at` | дата и время ответа |
| `product` | продукт |
| `product_version` | версия продукта |
| `platform` | `Web` / `Android` / `iOS` |
| `country` | страна |
| `user_segment` | сегмент пользователя |
| `score1`, `score2` | ответы UMUX-Lite, целые числа 1–5 |

## Установка

```bash
cd umux_pipeline
pip install -r requirements.txt
```

(Python 3.10+; внешние зависимости — pandas, matplotlib, а для опционального HTTP
API — fastapi/uvicorn.)

## Запуск (консоль)

```bash
python -m umux_pipeline.cli --input sample_data/raw_responses_sample.csv --outdir output
```

Можно передать сразу несколько файлов одной схемы (они объединяются перед очисткой,
например выгрузки за разные месяцы или из разных источников):

```bash
python -m umux_pipeline.cli --input jan.csv feb.csv mar.csv --outdir output
```

Результаты сохраняются в `--outdir` (по умолчанию `output/`):

- `report.html` — наглядная сводка (открывается в любом браузере)
- `clean_scored.csv` — валидные строки с `umux_score`
- `rejected_rows.csv` — отбракованные строки с `reject_reason`
- `rejection_reasons.csv`, `agg_by_product.csv`, `agg_by_product_version.csv`,
  `agg_by_month_product.csv`, `agg_by_segment.csv`, `overview.json`
- `pipeline.log` — полный лог запуска

Списки эталонных значений (например, другой каталог продуктов) переопределяются через
`--config my_config.json` — JSON-файл с любым подмножеством полей `PipelineConfig`
(см. `umux_pipeline/config.py`).

## Запуск (HTTP API — бонус)

```bash
uvicorn umux_pipeline.api:app --reload
```

- `POST /pipeline/run` — загрузка одного или нескольких `.csv` файлов (поле `files`,
  multipart), в ответ — сводка агрегатов в JSON плюс HTML-отчёт внутри ответа.
- `POST /pipeline/run/report` — та же загрузка, но в ответ сразу HTML-отчёт (можно
  открыть прямо в браузере).
- `GET /health` — проверка живости сервиса.

## Тесты

```bash
python -m pytest tests/ -v
```

Покрывают: нормализацию категориальных полей и исправление опечаток, разницу между
«неизвестное значение» и «жёсткая отбраковка», оценки вне диапазона, точные и
конфликтующие дубликаты, некорректные даты, формулу UMUX и сквозной прогон на
образце данных (включая проверку идемпотентности — повторный запуск даёт идентичные
агрегаты).

## Результат на образце данных

Из 1560 строк: **1383 валидны (88.7%)**, **177 отбракованы (11.3%)** — в основном
из-за пропущенных/некорректных оценок (119 строк) и точных дублей (51 строка).
Полная разбивка — в `output/report.html` после запуска пайплайна.

Готовый прогон на образце данных (для просмотра без установки/запуска) уже
закоммичен в [`output/`](output/) — в частности [`output/report.html`](output/report.html).

## Структура проекта

```
umux_pipeline/
  config.py        эталонные данные (канонические продукты/платформы/сегменты), пороги
  cleaning.py       загрузка, нормализация, валидация -> (valid, rejected)
  scoring.py        формула UMUX
  aggregation.py     свод по продукту / версии / месяцу / сегменту
  report.py          самодостаточный HTML-отчёт
  pipeline.py         оркестрация, логирование, идемпотентная запись результатов
  cli.py              консольная точка входа
  api.py              HTTP-точка входа на FastAPI (бонус)
tests/                тесты pytest
sample_data/           предоставленный образец выгрузки
```
